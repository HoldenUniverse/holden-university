Format & source-file findings (already investigated in this repl):

IntegrityOS already ships Three.js + @react-three/fiber + @react-three/drei (used by the Global Industrial Intelligence Map and Digital Twin explorer) — no new 3D engine needed.
The Perdido purchase came in many formats: .max (3ds Max 9/2014, proprietary, unusable without 3ds Max), .3ds/.dwg/.dxf (legacy/CAD exchange, not game-engine ready), .stl (~4.6MB, geometry only — no materials, no UVs, degenerate normals in the sample checked — usable only as a dumb collision/reference shape), .obj+.mtl (~59MB, has real materials/named groups — the best raw source available), .fbx (~30MB binary, has hierarchy/materials but is a proprietary format).
Recommendation: target glTF 2.0 binary (.glb) as the one runtime format, generated from the FBX or OBJ source (not STL/DWG/DXF/MAX). glTF is the standard for real-time web/WebGL/WebXR delivery, supports Draco/meshopt geometry compression, KTX2 texture compression, PBR materials, and named node hierarchies (critical for hotspot mapping) — and Three.js/R3F/drei load it natively via useGLTF/GLTFLoader.
This Repl has no Blender/FBX SDK/glTF conversion tooling installed. FBX→glTF or OBJ→glTF conversion must happen either (a) once, offline, using free Blender (recommended — the user or a helper runs Blender's built-in glTF exporter), or (b) by installing Blender as a system package in this Repl and running a headless (blender --background --python) conversion script as part of the admin pipeline in Step 1 below. Both are viable; the executor should attempt (b) first since it makes the pipeline repeatable for the ~19 future models, falling back to documenting the manual Blender export steps for the user if headless Blender doesn't run reliably in this environment.
Recommendation: prototype Perdido alone first.
