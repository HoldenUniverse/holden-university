Great question — and the architecture is actually already built to handle it. Here's how it works and where it's going:

The core idea: worlds are themed containers

Each Facility World is a physical location — an offshore spar, a refinery, a nuclear plant, a pipeline corridor — that naturally hosts the scenarios that belong there. Perdido currently has 9 hotspots pointing into real scenarios. But a single world can hold 50, 80, even 100+ hotspots.

The path to 1,900+ scenarios across all worlds:

World (planned)	Natural home for
Perdido Offshore Spar (live)	Rope Access, Piping, ISV/NDE scenarios
Refinery Process Unit	Piping, Turnaround, ICE, CME
Nuclear Power Station	NUC, ETA, ICE, HPE
Pipeline Corridor	PIP, FRE, CME, ERC
Offshore FPSO	ROB, DTC, UAV, RA
Petrochemical Complex	ISE / ICA flagship scenarios
Construction Site	CME, RAM, RAI, RAER
Control Room / OT Hub	ICE, DTC, HPE, ELA
Each world gets a 3D model + hotspots. Every hotspot is just a { pillar, scenarioId } reference — the resolver already handles every pillar universally, so no new plumbing is needed. You add a world, author its hotspots, and all 7 work-package documents, D-IMS records, CodeIQ questions, and mission briefs flow through automatically.

What "done" looks like:

~15 worlds × ~100 hotspots each = all 1,900 scenarios accessible as real physical locations you can walk through, investigate, and log findings in. The mission engine we just built (arrival card, radio chatter, investigation mode, finding logged toast) scales to every one of them with zero extra code — just data.

The bottleneck isn't code, it's 3D models. Each new world needs a .glb model file. The rest is authoring hotspot coordinates and mission briefs, which follows the same pattern as Perdido.

I think this is the right architectural direction because you're separating the engine from the content. Once the engine is stable, growth comes primarily from adding assets and data rather than writing new application logic.

Why this scales well

Most educational platforms create a lesson, then write separate quizzes, reports, simulations, and practical exercises.

Your architecture inverts that.

A single scenario can drive:

A 3D inspection mission
A CodeIQ exercise
A CertIQ practice question
A D-IMS work package
An inspection report
A competency assessment
A CPD activity
A debrief discussion
Analytics on decision-making

That means every new scenario has value across multiple parts of the platform instead of existing in only one place.

The main challenge becomes content production, not software architecture. The biggest investments are building or acquiring high-quality 3D facility models, authoring realistic missions, and creating the associated engineering documentation and evidence. If those are organized into reusable world packages and equipment libraries, the platform can continue expanding without requiring major changes to the underlying engine.
